description: "Get Seguridad App running locally and make your first authenticated API call."
---

<Steps>
  <Step title="Prerequisites">
    Make sure you have the following installed before you begin:

    - **JDK 21** â€” the project targets JVM 21. Verify with `java -version`.
    - **Gradle** â€” or use the included `./gradlew` wrapper (no separate install needed).
    - **PostgreSQL** â€” a running instance on port `5432`, or Docker to run one alongside the app.

    To start a PostgreSQL instance with Docker:

    ```bash
    docker run -d \
      --name seguridad-pg \
      -e POSTGRES_USER=postgres \
      -e POSTGRES_PASSWORD=postgres \
      -e POSTGRES_DB=seguridad_db \
      -p 5432:5432 \
      postgres:16
    ```
  </Step>

  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/VictorGHOSTH/seguridad-app.git
    cd seguridad-app
    ```
  </Step>

  <Step title="Set environment variables">
    The application reads three environment variables at startup. All have built-in defaults, but you should override them before going to production.

    | Variable | Default | Description |
    |---|---|---|
    | `DATABASE_URL` | `jdbc:postgresql://localhost:5432/seguridad_db` | Full JDBC connection URL for the PostgreSQL database |
    | `DATABASE_USER` | `postgres` | Database username |
    | `DATABASE_PASSWORD` | `postgres` | Database password |
    | `JWT_SECRET` | `my-secret-key-for-jwt-authentication-2024-seguridad-app` | HMAC-256 signing key for JWT tokens |
    | `RECAPTCHA_SECRET` | `6Lcj3ossAAAAAAFK0hIW8FrgfTQfeivriiYQUUEq` | Google reCAPTCHA v2 server-side secret key |

    Export them in your shell for the current session:

    ```bash
    export DATABASE_URL="jdbc:postgresql://localhost:5432/seguridad_db"
    export DATABASE_USER="postgres"
    export DATABASE_PASSWORD="postgres"
    export JWT_SECRET="replace-with-a-long-random-secret"
    export RECAPTCHA_SECRET="your-recaptcha-secret-key"
    ```

    <Warning>
      Never use the default `JWT_SECRET` in production. It is public knowledge. Generate a strong random value â€” at least 32 characters â€” before deploying.
    </Warning>
  </Step>

  <Step title="Run the server">
    <CodeGroup>

    ```bash Gradle
    ./gradlew run
    ```

    ```bash Docker
    ./gradlew buildFatJar
    docker build -t seguridad-app .
    docker run -p 8080:8080 \
      -e DATABASE_URL="jdbc:postgresql://host.docker.internal:5432/seguridad_db" \
      -e DATABASE_USER="postgres" \
      -e DATABASE_PASSWORD="postgres" \
      -e JWT_SECRET="replace-with-a-long-random-secret" \
      -e RECAPTCHA_SECRET="your-recaptcha-secret-key" \
      seguridad-app
    ```

    </CodeGroup>

    When the server starts successfully you will see:

    ```
    2024-12-04 14:32:45.584 [main] INFO  Application - Application started in 0.303 seconds.
    2024-12-04 14:32:45.682 [main] INFO  Application - Responding at http://0.0.0.0:8080
    ```

    On first start the server creates all database tables and seeds initial data: two profiles (`Administrador` and `Usuario Normal`), the `admin` user, eight modules, and three menus.
  </Step>

  <Step title="Make your first login call">
    Send a `POST /api/auth/login` request with your credentials and a reCAPTCHA token.

    ```bash
    curl -s -X POST http://localhost:8080/api/auth/login \
      -H "Content-Type: application/json" \
      -d '{
        "username": "admin",
        "password": "Admin123",
        "recaptchaToken": "<your-recaptcha-token>"
      }'
    ```

    A successful response returns a JWT and the user's full permission set:

    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiJ9...",
      "usuario": {
        "id": 1,
        "strNombreUsuario": "admin",
        "strCorreo": "admin@seguridad.com",
        "idPerfil": 1,
        "idEstadoUsuario": 1
      },
      "esAdministrador": true,
      "permisos": [
        {
          "idModulo": 1,
          "strNombreModulo": "Perfil",
          "bitAgregar": true,
          "bitEditar": true,
          "bitConsulta": true,
          "bitEliminar": true,
          "bitDetalle": true
        }
      ]
    }
    ```

    Store the `token` value â€” you need it for every subsequent request.

    <Note>
      reCAPTCHA validation is enforced on every login. For local development you can use a test token from Google's reCAPTCHA testing keys, or temporarily swap the secret key for a test key pair that always succeeds.
    </Note>
  </Step>

  <Step title="Call a protected endpoint">
    Pass the token as a `Bearer` value in the `Authorization` header:

    ```bash
    curl -s http://localhost:8080/api/auth/verify \
      -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..."
    ```

    The token verification endpoint returns:

    ```json
    {
      "valid": true,
      "userId": 1
    }
    ```

    Use the same `Authorization: Bearer <token>` header on all resource endpoints â€” users, profiles, modules, permissions, and menus.
  </Step>
</Steps>
